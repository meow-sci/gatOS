This program does exactly one thing: it tells a vessel's **flight computer** to point its main axis at
at the body it's orbiting (its **parent SOI**), and then it exits.

The flight computer takes care of the actual steering via available propulsion (RCS, etc.)

Uses only plain file reads and writes under `/sim`

## Pre-requisites

- python installed in gatOS: `apk add python3`

## The idea in one picture

To aim at the parent body we need to know which way the body _is_, and gatOS hands us that almost for
free thanks to the frame the data lives in.

`/sim/vessels/by-id/<id>/position/cci` is the vessel's position in **CCI** — _Celestial-Centered
Inertial_, the parent body's own reference frame:

- **origin** = the parent body's center,
- **+Z** = the body's north/spin axis, **+X** = the vernal point (a direction fixed in space),
- right-handed, in meters.

Because the origin sits at the body's center, `position/cci` is literally the arrow pointing _from
the body to the vessel_. So the arrow pointing the other way — _from the vessel to the body_ — is
just its negative. No trigonometry, no detour through latitude/longitude: **aim direction =
`-position/cci`**.

The vessel's nose/thrust axis is **body +X**. "Pointing at the body" therefore means: find the
rotation that carries body +X onto that aim direction, and hand it to the flight computer as a
quaternion in `ctl/attitude_target`.

## The program

Save this as `point-at-parent.py` in the guest and run `python3 point-at-parent.py <VesselId>`
(find ids with `ls /sim/vessels/by-id`).

```python
#!/usr/bin/env python3
"""point-at-parent.py — aim a vessel's nose/thrust axis at its parent body.

Usage (inside a gatOS guest shell):  python3 point-at-parent.py <VesselId>

The whole program is file I/O on /sim: we read one vector, do a little geometry,
and write one control file. That's it.
"""
import sys, math

vessel = sys.argv[1]                       # a vessel's *name* is its id, e.g. "Hunter"
base = f"/sim/vessels/by-id/{vessel}"      # everything about this vessel lives here

# --- talking to /sim --------------------------------------------------------
def read_vec(path):
    # Values in /sim are plain text. A vector is "x y z" + a newline.
    with open(path) as f:
        return [float(t) for t in f.read().split()]

def write(path, text):
    # A write to a ctl/ file actuates the game on the newline, and returns a
    # real Linux errno on failure (EACCES if control is off, EINVAL if the
    # value is bad, ...). So a failed setpoint raises OSError here.
    with open(path, "w") as f:
        f.write(text + "\n")

# --- tiny vector helpers (CCI is just a normal right-handed 3D frame) -------
def cross(a, b): return [a[1]*b[2] - a[2]*b[1], a[2]*b[0] - a[0]*b[2], a[0]*b[1] - a[1]*b[0]]
def norm(a):     return math.sqrt(sum(c*c for c in a))
def unit(a):     n = norm(a); return [c / n for c in a]

def from_rows(r0, r1, r2):
    """KSA's doubleQuat.CreateFromRotationMatrix (Shepperd's method), row-major.
    The three rows are the body X, Y, Z axes written in CCI; the result is the
    Body->CCI quaternion [x, y, z, w] the flight computer expects. This is a
    verbatim port of the game's own arithmetic — a generic quaternion library
    can use a different sign convention and steer the wrong way."""
    m00, m01, m02 = r0
    m10, m11, m12 = r1
    m20, m21, m22 = r2
    tr = m00 + m11 + m22
    if tr > 0.0:
        s = math.sqrt(tr + 1.0); w = 0.5 * s; s = 0.5 / s
        return [(m12 - m21) * s, (m20 - m02) * s, (m01 - m10) * s, w]
    elif m00 >= m11 and m00 >= m22:
        s = math.sqrt(1.0 + m00 - m11 - m22); inv = 0.5 / s
        return [0.5 * s, (m01 + m10) * inv, (m02 + m20) * inv, (m12 - m21) * inv]
    elif m11 > m22:
        s = math.sqrt(1.0 + m11 - m00 - m22); inv = 0.5 / s
        return [(m10 + m01) * inv, 0.5 * s, (m21 + m12) * inv, (m20 - m02) * inv]
    else:
        s = math.sqrt(1.0 + m22 - m00 - m11); inv = 0.5 / s
        return [(m20 + m02) * inv, (m21 + m12) * inv, 0.5 * s, (m01 - m10) * inv]

def any_orthogonal(v):
    # Some vector guaranteed perpendicular to v (used when roll is unconstrained).
    a = [1.0, 0.0, 0.0] if abs(v[0]) < 0.9 else [0.0, 1.0, 0.0]
    return unit(cross(v, a))

def body_to_cci(aim, roll_ref):
    """Build the Body->CCI quaternion that puts body +X (the nose/thrust axis)
    along `aim`. `roll_ref` only pins the otherwise-free spin about that axis."""
    x = unit(aim)                                   # body +X  -> where the nose points
    c = cross(x, roll_ref)
    y = any_orthogonal(x) if norm(c) < 1e-9 else unit(c)   # body +Y (roll ref, or free)
    z = unit(cross(x, y))                           # body +Z completes a right-handed triad
    return from_rows(x, y, z)

# 1. READ where we are. In CCI the origin is the parent body's center, so this
#    vector points from the body out to the vessel.
pos = read_vec(f"{base}/position/cci")

# 2. AIM. The body sits at the CCI origin, so the direction from the vessel *to*
#    the body is simply the negated position ("radially inward", toward the SOI).
aim = [-c for c in pos]

# 3. Turn that direction into a Body->CCI attitude quaternion. We pass `pos` as
#    the roll reference; since we're aiming straight down the radial line, roll
#    is unconstrained and gets picked for us — pointing a nose leaves spin free.
q = body_to_cci(aim, pos)

# 4. WRITE the setpoint. The onboard flight computer reads attitude_target and
#    steers there itself, so it stays correct even under time-warp. This write
#    is "solver-phase": it takes effect on the next physics step (~10 Hz), not
#    the instant the write returns.
write(f"{base}/ctl/attitude_target", "{} {} {} {}".format(*q))
print(f"{vessel}: flight computer now pointing at parent body")
```

## Run it

```sh
# in the gatOS guest shell
python3 point-at-parent.py Hunter
```

Watch the vessel swing around until its nose faces the planet. You just read live simulation state
and commanded an autopilot with nothing but `open()`.

:::note[This is a one-shot setpoint]
The program computes the aim direction _once_ and leaves. As the vessel travels along its orbit, the
true direction to the body keeps changing, so the fixed target slowly goes stale — perfect for a
first look, not for holding a lock. To keep pointing, wrap steps 1–4 in a loop that re-reads
`position/cci` and re-writes the target each tick (and pace it in sim time via `/sim/time/alarm`, so
it stays correct under warp).
:::

:::tip[Shortcut: let the autopilot name it]
This exact orientation — nose toward the parent — is a built-in mode. You can skip all the
quaternion math with a single line:

```sh
echo RadialIn > /sim/vessels/by-id/Hunter/ctl/attitude_mode
```

The named mode even tracks the body for you as you orbit. We built the quaternion by hand here to
show how CCI, the body +X thrust axis, and `ctl/attitude_target` fit together — the same machinery
you'd reuse for a _custom_ direction the built-in modes don't cover.
:::
